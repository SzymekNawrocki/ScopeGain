Podróż jednego żądania

Od kliknięcia w przeglądarce do danych z Postgresa — i z powrotem na kartę.

Po co ta lekcja. Na rozmowie padnie: „przeprowadź mnie przez architekturę". Wszystko, co zbudowałeś w warstwach 1–4 (Pydantic, routery, get_db, CORS, ORM), to przystanki na jednej trasie. Kto zna trasę, obroni każdy klocek. To jest ta trasa.

Prześledzimy jedno realne żądanie z Twojego repo: GET /portfolios — to, które zapełnia dashboard. Siedem przystanków. Przy każdym: co się dzieje, Twój kod, i „dlaczego tak".

Trasa w jednej linii

[Przeglądarka fetch] → CORS → [FastAPI] → router → [get_db sesja] → ORM → [Postgres]
        ↑                                                                              │
        └──────────────  Pydantic filtruje JSON  ←────────  obiekty  ←────────────────┘

Siedem przystanków

1Przeglądarka strzela fetchfrontend/app/page.tsx

Dashboard to Client Component. Po wejściu useEffect odpala się raz i woła API z przeglądarki użytkownika — nie z serwera.

// "use client" -> ten kod biegnie w przegladarce
useEffect(() => {
  getPortfolios().then(setPortfolios).catch(...);
}, []);  // [] = uruchom raz
Dlaczego to ma znaczenie: bo fetch leci z przeglądarki, wchodzi CORS (przystanek 2). Gdybyś pobierał dane w Server Component, żądanie szłoby serwer→serwer i CORS by nie istniał.
2Przeglądarka pyta: „czy wolno?" (CORS)backend/main.py

Front (:3000) i API (:8000) to różne pochodzenia. Przeglądarka domyślnie blokuje taki fetch, dopóki serwer nagłówkiem nie powie „ufam".

app.add_middleware(
  CORSMiddleware,
  allow_origins=["http://localhost:3000"],  # biala lista frontu
  allow_methods=["*"], allow_headers=["*"],
)
Dlaczego to ma znaczenie: CORS to ochrona przeglądarki, nie „błąd backendu". Serwer zwraca nagłówek Access-Control-Allow-Origin — bez niego przeglądarka wyrzuca odpowiedź.
3FastAPI dopasowuje trasę do routerabackend/routers/portfolios.py

Żądanie trafia do aplikacji. main.py wpiął router z prefiksem /portfolios, więc GET /portfolios ląduje w tej funkcji:

@router.get("", response_model=list[PortfolioRead])
def list_portfolios(db: Session = Depends(get_db)):
    return db.scalars(select(Portfolio)).all()
Dlaczego to ma znaczenie: router = temat w osobnym pliku. main.py nie zna szczegółów — składa apkę z klocków. Dodasz auth → nowy router, reszta nietknięta.
4Depends dostarcza sesję bazybackend/database.py

Zanim funkcja się wykona, FastAPI widzi Depends(get_db) i wstrzykuje świeżą sesję. finally gwarantuje jej zamknięcie — nawet gdy coś wybuchnie.

def get_db():
    db = SessionLocal()
    try:    yield db   # oddaj sesje endpointowi
    finally: db.close()   # zawsze posprzataj
Dlaczego to ma znaczenie: to dependency injection. Bez zamykania sesji połączenia wyciekają i baza się zatyka. Framework robi to za Ciebie, przy każdym żądaniu.
5ORM tłumaczy Python na SQLbackend/models.py

select(Portfolio) to nie SQL — to Python. SQLAlchemy (ORM) zamienia to na SELECT * FROM portfolios i odpala na Postgresie.

class Portfolio(Base):
    __tablename__ = "portfolios"
    id: Mapped[int] = mapped_column(primary_key=True)
    positions: Mapped[list["Position"]] = relationship(...)  # relacja 1—wiele
Dlaczego to ma znaczenie: piszesz w jednym języku (Python), nie w dwóch. Ta relationship sprawia, że portfel „wie" o swoich pozycjach — kluczowe w przystanku 7.
6Postgres zwraca wiersze → obiektydocker: scopegain-db

Baza w kontenerze Dockera oddaje wiersze. ORM pakuje je z powrotem w obiekty Portfolio (każdy z listą Position). Dane przeżywają restart — to nie pamięć.

Dlaczego to ma znaczenie: to jest różnica między „skryptem" a „systemem". Stan żyje w bazie, aplikacja jest bezstanowa — możesz ją zrestartować, dane zostają.
7Pydantic filtruje odpowiedź → JSON → kartaschemas.py → page.tsx

response_model=list[PortfolioRead] przepuszcza obiekty przez schemat: do klienta lecą tylko pola ze schematu (z zagnieżdżonymi pozycjami), jako JSON. Front renderuje kartę.

class PortfolioRead(BaseModel):
    id: int; name: str
    positions: list[PositionRead] = []
    model_config = {"from_attributes": True}  # czytaj z obiektu ORM
Dlaczego to ma znaczenie: schemat to kontrakt i filtr wyjścia. Nawet gdy obiekt z bazy ma pola wewnętrzne, klient dostaje tylko to, co zadeklarowałeś. Tu koszt wejścia = 10·150 + 5·300.
